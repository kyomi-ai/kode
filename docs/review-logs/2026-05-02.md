### drag-and-drop bugfix — non-extension block position adjustment

**Review type**: bugfix review
**Result**: 0 🔴, 0 🟡, 1 🟢 — APPROVED
**Files**: `kode-leptos/src/wysiwyg/tree_editor.rs` (~10 lines changed in the pointermove closure)

**Bug**: When dragging chart blocks to positions near non-extension blocks (paragraphs, headings), the chart disappeared — it was being inserted INSIDE the other block instead of between blocks.

**Root cause**: Non-extension blocks store content positions in `data-pos-start`/`data-pos-end` (1 token inside the block). The pointermove handler was using these raw positions as drop targets directly, so "before a paragraph" would target the paragraph's content_start (inside it).

**Fix verification**:

- `is_extension = el.has_attribute("data-kode-extension")` correctly detects extension vs. non-extension blocks.
- `block_start = raw_start.saturating_sub(1)` correctly recovers the block-open token position; `saturating_sub` handles the position-0 edge case.
- `block_end = raw_end + 1` correctly recovers the block-close token position.
- Drag handles are mounted only on extension blocks (`mount_drag_handles` at line 1692 queries `div.kode-extension-block[data-kode-extension]`), so `source_start`/`source_end` are always block-level positions — no adjustment needed on the source side.
- All downstream logic (distance comparison, source-range guard, `move_block` call) is untouched.

Use `raw_end.saturating_add(1)` for consistency and defensive symmetry. Overflow is not a realistic risk but the asymmetry is a minor style inconsistency. |
